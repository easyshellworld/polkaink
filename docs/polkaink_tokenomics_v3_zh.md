## 1. 设计理念

PolkaInk v3 综合了两个前版本的优点：来自 v2 代币经济模型的精确经济激励结构与质押参与机制，以及来自 v1 开发文档的健壮治理安全设计（TimelockController、委员会选举、参与率门槛）。最终形成一个公平、抗操纵、可持续的治理模型。

**核心原则：**

- **质押参与制** — 88 DOT 质押作为入场凭证，利益与风险深度绑定
- **纯粹民主投票** — 无 NFT 等级、无层级特权、无个人一票否决
- **贡献加权激励** — Historian（写作贡献）与 Guardian（委员会任职）NFT 通过真实链上行为自然积累权重
- **锁仓时间对齐** — 更长的锁仓承诺换取更高的话语权
- **Archive Council 集体守护** — 7 名民选委员共同持有 veto 权，公开透明、任期制、可罢免
- **TimelockController 信任锚** — 不可升级的执行延迟机制，系统的终极安全保障，无任何管理员绕过路径

## 2. NFT 系统

NFT 系统精简为两种类型。无等级 OG 排名，无团队空投特权 NFT，无任何特殊阶层。所有 NFT 均通过链上行为获得。

### 2.1 Historian NFT（历史作者徽章）

| 属性 | 说明 |
|------|------|
| 标准 | ERC-721，投票期间可自愿设为 Soulbound |
| 铸造触发 | 版本提案成功合并后自动 mint |
| 可叠加 | 是 — 每个成功提案获得一枚，无数量上限 |
| 过期 | 永久有效 |
| 可转让 | 默认可转让；投票期间可自愿锁定为 Soulbound |
| 加成效果 | 累计叠加 — 详见第 3 节公式 |
| Token URI | 全链上 SVG 生成（无 IPFS 依赖） |
| 视觉风格 | Polkadot 粉色系，◎ 符号，自增编号徽章 |

> **设计说明：** Historian NFT 的叠加反映真实贡献深度。采用 ln() 曲线意味着早期 NFT 边际价值更高，但持续积累仍有意义，兼顾早期参与者与长期贡献者。无空投 NFT 即无特权阶层——每枚 NFT 都是赚来的。

### 2.2 Guardian NFT（守护者徽章）

| 属性 | 说明 |
|------|------|
| 标准 | ERC-721，完全 Soulbound（不可转让） |
| 铸造触发 | 当选 Archive Council 委员后自动 mint |
| 每地址数量 | 1 枚（绑定单个委员席位） |
| 过期 | 6 个月任期结束或被罢免时自动 deactivate |
| 加成效果 | 激活状态下投票权重 +0.20 |
| Veto 权 | 是 — 作为 7 人集体的一票 |
| 视觉风格 | Polkadot 紫色系，盾牌图标，任期信息链上嵌入 |

## 3. 投票权重公式

每位活跃成员基础权重为 `1.0`。贡献与承诺附加正向加成。无负向乘数，Archive Council 集体机制之外不存在单人 veto 权。

### 3.1 公式

```
weight  =  has_active_member × (1 + boost)

boost   =  B_historian  +  B_guardian  +  B_lock

B_historian  =  0.40 × ln(1 + historian_count) / ln(21)
B_guardian   =  0.20 × has_active_guardian_nft
B_lock       =  0.40 × ln(1 + lock_months) / ln(25)
```

### 3.2 加成分项说明

| 组件 | 最大加成 | 获取方式 | 公式基础 |
|------|---------|---------|---------|
| B_historian | +0.40 | 撰写并通过提案 | ln 曲线，20 枚 NFT ≈ 达到上限 |
| B_guardian | +0.20 | 当选 Archive Council | 二元值：Guardian NFT 激活 = +0.20 |
| B_lock | +0.40 | 更长质押锁仓期 | ln 曲线，24 个月 = 达到上限 |
| **总最大加成** | **+1.00** | — | **最大权重 = 2.00** |

> **为什么最大权重 = 2.00？**
> 门槛 `T = 2.0` 意味着即使是最大权重的单个用户也无法单独通过提案。至少需要两位活跃成员同意。这在奖励深度贡献者的同时，保留了真正的社区共识要求。对比 v2 最大权重 2.35 允许单人通过提案的问题。

### 3.3 典型用户画像示例

| 用户画像 | B_hist | B_guard | B_lock | 总加成 | 权重 |
|---------|--------|---------|--------|--------|------|
| 新成员，3 个月锁仓 | 0 | 0 | 0.13 | 0.13 | 1.13 |
| 5 篇提案，6 个月锁仓 | 0.23 | 0 | 0.16 | 0.39 | 1.39 |
| 10 篇提案，12 个月锁仓 | 0.29 | 0 | 0.22 | 0.51 | 1.51 |
| 委员会成员，12 个月锁仓 | 0 | 0.20 | 0.22 | 0.42 | 1.42 |
| 10 篇提案 + 委员，24 个月 | 0.29 | 0.20 | 0.40 | 0.89 | 1.89 |
| **最大值：20 篇提案 + 委员，24 个月** | **0.40** | **0.20** | **0.40** | **1.00** | **2.00** |

## 4. 提案通过条件

提案通过须**同时满足以下三个条件**：

```
条件一 — 票数阈值
  Σ(vote_i × weight_i) > T = 2.0
  [赞成 = +weight，反对 = −weight，弃权 = 0]

条件二 — 参与率门槛
  totalParticipantWeight / snapshotTotalWeight > 5%

条件三 — Archive Council 未触发 veto
  委员 veto 数量 < 4（7 人委员会中）
```

### 4.1 为什么 5% 参与率至关重要

5% 参与率门槛（v2 中完全缺失）对治理合法性至关重要。没有它，无论总成员规模多大，只需 2 人参与即可通过提案：

- 小规模协调团体无法悄无声息地捕获治理权
- 提案需要可见的社区参与，而非仅仅依赖其他人的缺席
- 冷启动保护：Phase 1 参与率门槛从 3% 开始，Phase 2 升至 5%

### 4.2 场景示例

| 场景 | 分数 | 参与率？ | Veto？ | 通过？ |
|------|------|---------|-------|-------|
| 仅作者（w=1.39） | +1.39 | ❌ 不足 | — | ❌ |
| 作者 + 1 成员赞成 | +2.52 | 取决于总量 | — | 视情况 |
| 10 成员全赞成（基础权重） | +11.3 | ✅ 通过 | — | ✅ |
| 10 赞成，3 委员反对 | +9.0 | ✅ 通过 | ❌ 不足 4 票 | ✅ |
| 10 赞成，4 委员反对 | +7.2 | ✅ 通过 | **VETO** | ❌ |
| 作者 + 2 赞成，参与率达标 | +3.91 | ✅ 通过 | — | ✅ |

## 5. 质押与成员资格

通过质押恰好 88 DOT 并选择锁仓期来获得成员资格。锁仓到期后全额返还（或以 10% 罚金提前解锁）。无等级，无 OG 配额，无特权空投接收者。

### 5.1 质押参数

| 参数 | 数值 | 备注 |
|------|------|------|
| 质押金额 | 88 DOT | 固定值；可通过 ParameterChange 提案调整 |
| 锁仓选项 | 3 / 6 / 12 / 24 个月 | 锁仓越长，B_lock 加成越高 |
| 提前解锁罚金 | 10% | 罚金进入 Treasury |
| 成员状态 | 锁仓期间有效 | 锁仓结束后过期 |
| 过期后 NFT | 纪念性（无投票权） | 链上永久记录 |
| 多地址 | 允许 | 每个地址独立；通过参与率门槛防女巫 |
| 重新质押 | 过期后立即可操作 | 新锁仓期重新开始 |

### 5.2 锁仓时长与加成对照

| 锁仓时长 | B_lock | 权重（无其他加成） |
|---------|--------|-----------------|
| 3 个月 | 0.13 | 1.13 |
| 6 个月 | 0.16 | 1.16 |
| 12 个月 | 0.22 | 1.22 |
| 24 个月 | 0.40 | 1.40 |

```
B_lock = 0.40 × ln(1 + lock_months) / ln(25)   // ln(25) ≈ 3.22
```

## 6. Archive Council（档案守护委员会）

Archive Council 是 PolkaInk 的最后一道伦理防线。它取代了 v1 中由团队指定的委员会，以及 v2 中将过多权力集中于单人的 OG Gold 个人否决机制。委员会完全由社区选举产生，权力集体行使，任期制约束。

### 6.1 核心原则

- **仅 veto，不主导** — 委员会无法主动发起内容提案或预防性封锁，只能对已通过社区投票的提案行使否决权
- **集体门槛** — 7 人中 4 人投反对票才能触发 veto；不存在单人否决权
- **全部记录上链** — 每次 veto 永久记录在链，附必填书面理由（≥ 50 字节）
- **无固定薪酬** — Guardian NFT 提供声誉激励；消除经济腐败动机
- **社区可罢免** — DAO 75% 超级多数投票可随时罢免任何委员
- **6 个月任期轮换** — 防止权力固化；每届任期后重新选举

### 6.2 委员资格要求（候选人准入）

| 要求 | 数值 | 用途 |
|------|------|------|
| Polkadot 账户存活时间 | ≥ 12 个月 | 防止新建女巫账户 |
| DOT 持有量 | ≥ 100 DOT | 利益绑定要求 |
| 链上违规记录 | 无活跃违规 | 要求清白记录 |
| 公开联系方式 | Discord 或 Element | 社区问责渠道 |
| 提名方式 | 7 天窗口期内自我提名 | 开放、无需许可的流程 |

### 6.3 选举流程

```
任期结束前 14 天
  → 任何成员均可发起选举

7 天提名期
  → 符合条件的候选人链上自我提名

7 天投票期
  → 投票权重 = DOT 余额 + Historian NFT 加成

结果执行
  → 得票最高的 7 人当选
  → 候选人不足 7 人时，委员会规模相应缩减
  → 旧 Guardian NFT 失效 → 新 Guardian NFT 铸造
```

### 6.4 Veto 机制细节

| 方面 | 设计 |
|------|------|
| 窗口期 | 提案进入 TimelockQueued 状态后 48 小时内 |
| 门槛 | 7 人委员会中 4 人投反对票 |
| 理由必填 | ≥ 50 字节链上文字（永久记录） |
| 适用范围 | 所有提案类型，争议复议（re-vote）流程除外 |
| 反 veto | DAO 80% 超级多数可推翻 veto 并重启提案 |
| 对质押的影响 | 被 veto 的提案：50% 质押扣罚 → Treasury |
| 合理 veto 范围 | 虚假历史内容、恶意合约升级、法律风险内容、歧视性内容 |

## 7. 争议与冻结机制

任何活跃成员均可举报已通过的文档。相较于 v2 硬性限制 2 次举报，v3 在冷却期后允许第三次争议，防止争议内容被永久固化。

### 7.1 举报触发条件

```
举报数 ≥ max(3, floor(原始反对票数 × 1.5))
```

该公式保证比例性：争议越大的提案需要更多举报才能触发冻结，而全票通过的内容仍保留最低 3 次举报门槛。

### 7.2 流程

| 阶段 | 时长 | 规则 | 结果 |
|------|------|------|------|
| 冻结 | 72 小时 | 文档冻结，禁止编辑 | 自动进入复议投票 |
| 复议投票 | 72 小时 | 1 成员 = 1 票（无加成，无 Council veto）；参与门槛 ≥ 5 人 | 赞成 > 反对 → 维持；否则 → 撤销 |
| 撤销后冷却 | 30 天 | 相同内容不得重新提交 | 冷却期后可重新提交 |

### 7.3 举报次数限制（相较 v2 的改进）

| | v2 | v3 |
|--|----|----|
| 每文档最大举报次数 | 2 次（硬性上限） | 180 天窗口内 2 次；冷却期后可追加 1 次 |
| 达到上限后 | 永久不可挑战 | 180 天后如有新证据，可发起第三次举报 |
| 设计依据 | 防止骚扰 | 防骚扰的同时，允许纠正新发现的问题 |

## 8. 提案生命周期

所有提案遵循相同的生命周期，通过相同的安全门控。不同类型提案有不同的最低质押要求。

### 8.1 提案类型与最低质押

| 类型 | 最低质押 | 用途 |
|------|---------|------|
| VersionUpdate | 5 DOT | 提交或更新历史文档版本 |
| ParameterChange | 20 DOT | 修改治理参数 |
| CouncilElection | 10 DOT | 发起 Archive Council 选举 |
| TreasurySpend | 10 DOT | 申请 Treasury 资金支出 |
| UpgradeContract | 50 DOT | 提交合约升级（需附审计报告链接） |
| DisputeProposal | 5 DOT | 挑战已通过的文档 |

### 8.2 生命周期阶段

```
已提交
  → 投票中（7 天投票期）
    → 已通过 / 已拒绝

已通过
  → TimelockQueued（48 小时 Archive Council 审查窗口）
    → 已执行（未触发 veto）
    → 已 veto（4/7 委员投反对票）
```

### 8.3 质押结果

| 结果 | 质押去向 |
|------|---------|
| 提案通过 | 100% 返还 + 按比例 Treasury 奖励 |
| 投票否决 | 30% 扣罚 → Treasury；70% 返还 |
| Council veto | 50% 扣罚 → Treasury；50% 返还 |
| 提案人主动取消（Pending 状态） | 100% 返还（鼓励先讨论再提交） |

## 9. 财库与经济模型

### 9.1 资金来源

| 来源 | 金额 | 适用阶段 |
|------|------|---------|
| 88 DOT 成员质押（托管，不消耗） | 动态 | 全阶段 |
| 提前解锁罚金（10%） | 动态 | 全阶段 |
| 质押扣罚（30%–50%） | 动态 | 全阶段 |
| Polkadot Treasury Grant | 5 万–10 万 DOT | Phase 0–1 |
| API 企业访问费 | 按调用量计费 | Phase 3+ |
| DeFi 收益（≤20% 财库部署） | 估算年化 5–8% | Phase 3+ |
| 社区捐赠 | 无上限 | 全阶段 |

### 9.2 奖励分配（每个合并的提案）

```
总奖励池（DAO 定期投票从 Treasury 划拨）
  │
  ├── 70% → 作者（提案提交人）
  │          合并时直接转账 + 触发 Historian NFT 铸造
  │
  └── 30% → DAO Treasury
             ├── 50% → 补充后续奖励池
             ├── 30% → 协议运营（前端托管、索引节点）
             └── 20% → Archive Council 运营基金
```

### 9.3 财库治理

- 所有财库支出均需通过 `TreasurySpend` 提案，走标准投票流程
- DeFi 流动性部署需额外 DAO 授权；单次部署上限 ≤ 总财库的 20%
- 每笔独立部署上限 ≤ 单个提案中总财库的 10%
- TimelockController 管控所有财库出账 — 团队无任何直接访问权限

### 9.4 冷启动策略

| 措施 | 详情 |
|------|------|
| 创世 Historian NFT 空投 | 向 100 位 Polkadot 生态公认贡献者空投 — 纯荣誉性，无任何投票特权 |
| Phase 1 降低质押门槛 | 最低质押临时降至 1 DOT；参与率 3%；Phase 2 恢复正常 |
| Gas 补贴 | 前 50 个成功提案：Treasury 退还 50% Gas 费用 |
| 合作伙伴接入 | 从第 1 天起与 Polkassembly、Subscan、SubSquid 建立数据互通 |
| Grant 锚定 | Polkadot Treasury Grant 作为基础资金，降低早期对手续费收入的依赖 |

## 10. 合约安全

### 10.1 升级安全设计（继承自 v1）

`TimelockController` 不可升级，是系统的信任锚。任何合约升级、财库支出或参数修改均无法绕过它。

```
步骤 1  开发者提交 UpgradeContract 提案
        → 最低质押 50 DOT + 说明中必须包含审计报告链接

步骤 2  社区投票：7 天
        → 需要 60% 赞成 + 5% 参与率

步骤 3  Archive Council 审查：48 小时 Timelock 窗口
        → 可 veto 恶意升级（如隐藏管理员后门）

步骤 4  Timelock 到期后任何人均可执行
        → TimelockController.execute()
        → ProxyAdmin.upgradeAndCall(proxyAddr, newImpl, initData)

步骤 5  新实现生效
        → 链上发出 Upgraded(newImplementation) 事件
        → 前端自动检测新 ABI
```

> ⚠️ **紧急回滚** 需要提交新的 `UpgradeContract` 提案，走完整治理流程。不存在快速通道。

### 10.2 角色权限

| 角色 | 持有者 | 权限 |
|------|--------|------|
| GOVERNANCE_ROLE | GovernanceCore | 合并提案、铸造 Historian NFT |
| COUNCIL_ROLE | ArchiveCouncil | 在 GovernanceCore 中触发 veto |
| WRITER_ROLE | PolkaInkRegistry | 写入 VersionStore |
| AUTHOR_MINTER_ROLE | PolkaInkRegistry | 铸造 Historian NFT |
| GUARDIAN_MINTER_ROLE | ArchiveCouncil | 铸造 / 注销 Guardian NFT |
| SPEND_ROLE | **仅 TimelockController** | 执行 Treasury 支出 |
| UPGRADE_ROLE | **仅 TimelockController** | 通过 ProxyAdmin 执行合约升级 |

> **关键：无管理员覆盖路径。** 与 v2 不同（v2 中管理员可直接撤销 OG Gold NFT），v3 不存在任何管理员覆盖路径。Guardian NFT 注销需要通过 DAO 75% 超级多数投票，经 TimelockController 执行。团队在部署后无任何特殊密钥、后门或特权地址。

## 11. 防女巫攻击设计

多地址行为明确允许（与 v2 一致），但通过结构性设计加以缓解，而非一味禁止（因为链上禁止无法强制执行）。

| 攻击向量 | v3 缓解措施 |
|---------|-----------|
| 多地址质押刷票 | 每个地址 88 DOT — 额外地址有较高经济成本 |
| 通过多地址提案刷 Historian NFT | 每个提案需 5 DOT 质押 + 社区参与率门槛；刷量成本线性增长 |
| 操纵参与率 | 基于总权重快照的 5% 门槛，防止少数人捕获 |

## 13. v1 / v2 → v3 设计决策对照

| 前版本问题 | v3 解决方案 |

| v2：单个 OG Gold 持有者可独自否决任何提案 | 替换为委员会集体 veto（需 4/7 门槛） |
